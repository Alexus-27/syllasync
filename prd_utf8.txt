1. Project Overview



Nowadays, one of the most important ways to control students??education and performance is following the attention system, on time, making assignments, and without limitations and extra difficulties have the possibility to ask the professor about missing topics and other things. 



Knowing these requirements for ideal performance during the semester in Woosong University can lead to chaos in controlling the schedule, which assignments students have and how to communicate, and when the professor can notice students??messages. 





SyllaSync

 is a web and (extensible in the next step) mobile platform connecting professors and students through a structured syllabus, schedule, and communication system. All course information lives in one place ??accessible on a browser and smartphone.

The MVP of SyllaSync is a streamlined web platform designed to centralize course schedules, assignments, and communication between professors and students at Woosong University.









1.1 Problem Statement



Students often miss schedule changes, homework deadlines, and lesson notes because information is scattered across emails, group chats, and paper handouts. Professors lack a structured channel to broadcast updates and track student awareness. SyllaSync centralises everything into one platform.





1.2 Goals





Give professors a single dashboard to manage all course communications





Let students view their full academic schedule in a unified calendar





Enable per-course messaging between professors and students





Deliver real-time notifications for schedule changes and new homework





Work seamlessly on both web browsers and mobile devices





1.3 Scope ??v2.0 Includes





Professor dashboard: post schedule changes, homework, lesson notes, announcements





Student dashboard: unified calendar view, homework feed, notes viewer





Per-course chat between professors and enrolled students





Push notifications for changes and new content





Course enrollment via invite code





Web (React 18). Extensile to hybrid with Mobile (React Native / Expo) in the next step

3.5 Key Use Case Flow ??Schedule Change



The following describes the end-to-end flow when a professor reschedules a lesson:





Step 1. Professor opens Course Editor and taps the lesson to reschedule.





Step 2. Professor updates date, time, and room, then confirms the change.





Step 3. Web/App sends PUT /lessons/:id to the API with the new values.





Step 4. Backend updates the database and queues a push notification for all enrolled students.





Step 5. Students receive the push notification, open the app, and see the updated calendar. (The Student Calendar reflects the change immediately (via polling or lightweight real-time service).

6. Requests &amp; Suggestions



To ensure the successful development of 

SyllaSync

 within the June 15 deadline, the team requests the following support from supervisors, stakeholders, and the university.





Request 1 ??Access to Real Course Data for Testing



Access to sample course schedules, lesson structures, and homework formats from actual university courses will allow the team to validate the calendar and homework feed with realistic data, improving accuracy before the final demo.





Request 2 ??Regular Feedback from Supervising Professor



Biweekly check-ins, or async feedback on progress reports, from the supervising professor will ensure the system's features align with actual teaching workflows and institutional expectations. This prevents late-stage redesigns.





Request 3 ??Clarification of Expected Deliverables



Clear documentation of what the final submission should include ??live demo, code repository, written report, or presentation deck ??will help the team allocate the final two weeks appropriately without last-minute surprises.





Request 4 ??Student User Testing Volunteers



Access to 5??0 student volunteers willing to test the application during Week 9 user acceptance testing will provide realistic usage patterns and surface usability issues that the development team cannot identify internally.





Request 5 ??Guidance on Vibe Coding Boundaries



The team uses AI-assisted (vibe) coding for boilerplate and scaffolding. Clarity on how much AI-generated code is acceptable in the final submission, and whether it should be disclosed, will help the team code responsibly and avoid academic integrity concerns.





Request 6 ??Free-Tier Hosting Access or University Server



Hosting the backend and database on Railway or Render free tier may introduce cold-start latency during the demo. If the university can provide a small server or cloud credits, a more stable and reliable demo experience is possible.
